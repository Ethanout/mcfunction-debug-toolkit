## 目标

将 `.mcfunction` 文件中的连续 `#!` 注释行转换为简短的运行时注释行
调试语句。输出必须使用本机 Minecraft `Component` 行为并且是
发送给所有玩家，同时为本地 AI 桥保留结构化副本。
无效指令在重新加载期间会发出警告一次，并且会被忽略，但不会阻止
加载或运行函数的其余部分。

## 修复语法决策

- 通用形式：`#! plaintext {content[:format]} plaintext`。
- 连续的 `#!` 行可以形成一个指令，而大括号保持不平衡。
- 正常的非 `#!` 行结束收集。不完整的指令会被警告并
  丢弃；普通线路永远不会被消耗。
- 文字格式/明文大括号使用 `\{` 和 `\}`。 `\\`、`\"`、`\n` 和
  `\t` 在适用的情况下受支持转义。
- 查询调度：
  - 一种代币：`name`、`dim|dimension`、`pos|position`、`rot|rotation`、
    `anch|anchor`、`fname|function_name`、`fstack|function_stack`；
  - 两个标记：记分板持有者/选择者加上目标；
  - 显式：`storage <id> <nbt-path>`、`entity <selector> <nbt-path>`、
    `block <x> <y> <z> <nbt-path>`。
- `feedback|fdb`、`self`、`sname` 和 `self_name` 不存在。
- `name` 遵循与 `/say` 相同的命令源显示名称行为。
- `dimension` 呈现客户端本地化组件，并具有命名空间 ID
  悬停文本；结构化事件始终保留命名空间 ID。
- `position` 和 `rotation` 默认为 Java 的最短往返数字文本。
- `anchor` 是 `feet` 或 `eyes`。
- `fname` 是当前功能 ID。 `fstack` 是当前同步
  函数栈；预定函数启动一个新的根堆栈。

## 集合和格式决策

- `...` 每个列表格式范围最多可能出现一次，并且仅在该范围的末尾出现
  范围，后跟可选的空格，然后是可选的 `/strip` 或
  `/no_strip`。在该范围内，之后不能发生任何其他事情。
- 重复模式是当前格式开头的确切内容
  范围为 `...` 令牌。嵌套格式范围独立重复。
- `/strip` 是默认值。它仅截断最终重复的未使用的尾部
  模式。重复范围之外的固定文本永远不会被删除。
- 如果没有 `...`，集合查询最多接受一个运行时值。超过
  一个值会丢弃正常指令输出，在游戏中呈现短暂的红色错误
  对于 `@a`，并发出包含完整详细信息的限速结构化诊断。
- 空的本机动态内容遵循普通文本组件行为。
- 评分行为：
  - `{}` 为分值；
  - 项目字段：`{score}`、`{name}`、`{display_name}`、`{holder}`；
  - `name`和`display_name`是实体/玩家显示组件；
  - `holder` 是记分牌持有人的确切名称；
  - 没有客观分数的实体被跳过；
  - `*` 遵循普通记分牌持有者的行为，包括离线/假名字；
  - 默认单项结果为得分；默认多重结果是显示名称加
    得分，持有者姓名位于悬停文本中。
- 数据行为：
  - `{}`和`{value}`是当前NBT值；
  - 字段包括 `{entity}`、`{index}`、`{entity_index}`、`{global_index}`；
  - 在内部保留源/值层次结构；第一个渲染器可能会暴露
    展平叶视图而不丢弃层次结构。
- 复合项目格式写为`{"..."}`，例如
  `{"{display_name}: {score}"}`。
- 数字格式子集：`d`、`f`、`e`、`g` 和自定义 `p`（有效数字、
  从不指数），具有精度、宽度、对齐（如果可行）和零
  填充。应用于非数字的数字格式将被忽略，原始格式
  价值被呈现； AI 诊断可能会注意到这一点。

## 限制

- 每个指令最多 256 个呈现的集合叶。
- 每个指令最多 32,000 个渲染字符。
- 最多 64 个嵌套数据/模板级别。
- 运行时重复警告由函数 ID、行和错误代码键入，并且
  速率有限。
- 运行时错误使用简短的人工消息，例如
  `[#! test:main:18] expected one value, got 4`；详细的选择器/路径/错误数据
  属于日志和人工智能诊断而不是聊天线。
- 截断附加可见标记并在事件中设置 `truncated=true`。

## 架构

1. 在原版丢弃注释行之前拦截 Minecraft 函数加载。
2. 收集连续的 `#!` 行并在数据包重新加载期间解析它们。
3. 将有效的不可变指令 AST 存储在重载范围的注册表中，该注册表由
   稳定的指令ID；将每个有效块替换为一个已注册的合成块
   命令。忽略无效块并记录源映射警告。
4.通过Fabric命令注册来注册合成命令。它必须
   使用传入的 `CommandSourceStack` 执行，而不是新创建的源。
5. 通过执行跟踪服务器线程上的同步函数调用堆栈
   帧深度。当函数排队时，将其 ID 写入 `frame.depth + 1` 并
   清除该深度或更深处的条目。渲染时，只读条目最多
   当前帧深度。 Vanilla 函数执行是基于队列的，所以
   mixin 不得仅在 `CallFunction.execute` 返回时弹出。
6. 渲染到本机 `MutableComponent` 节点。重用普通选择器、NBT 路径、
   坐标、资源位置、文本组件分辨率和 `/tellraw`
   任何可用的交付行为。不要过早压平普通弦。
7. 将解析后的组件发送给每个玩家。还附加一个有界结构化
   包含函数、源代码行、组件/普通后备的调试事件，
   调用堆栈、截断和诊断。
8. 添加经过身份验证的 HTTP 端点以轮询调试事件并重新加载
   诊断。添加相应的TypeScript协议类型和MCP工具。

## 停止条件

- 如果没有安全的函数加载拦截点可以保留原版，则停止并报告
  经过多次验证尝试后进行解析。
- 停止并报告，而不是近似普通的tellraw行为，因为它不能
  从当前 26.2 代码/文档中确认。
- 切勿修改旧的 `minecraft-mod-mcp` 项目。
